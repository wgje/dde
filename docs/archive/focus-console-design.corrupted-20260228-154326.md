# 专注控制台（Focus Console）策划案

> **版本**：v3.1  
> **日期**：2026-02-28  
> **状态**：Active — v3.1 逻辑闭环精确化：三维推荐阵列时间约束 + 空集三场景处理 + 倦怠计数器持久化 + 全量级联缺口二级闭合  
> **前置依赖**：`focus-mode-design.md` v1.4、`parking-dock-modular-design.md` v1.4、`state-overlap-design.md` v10.0  
> **核心隐喻**：任务是船，停泊坞是港，专注控制台是舰队指挥舱——同一时刻只驾驶一艘，但港中候命、随时就绪。

> **v3.1 升级摘要**：在 v3.0 基础上对 40 项 G 缺口的级联新缺口(NG)进行精确闭合。核心变更：①§4.2.4 三维推荐阵列各组增加 T_est≤W_main 时间约束；②空集处理拆分为三场景（A≥30min 弹出大任务卡·属性可重叠、B<30min 留白期代替、C 部分空正常显示）；③§14 服务命名强制对齐 v3.0（comboSelectTasks/backupTasks）；④§2.5 FocusSessionState 增加 highLoadCounter + burnoutTriggeredAt 字段；⑤§7.8 Level 2 补全倦怠检测算法（2h 滑动窗口×3 高负荷触发）；⑥§0.1 逻辑缺口表 40 条全部标注修改版本号 + 二级级联缺口列。

## 二、停泊坞入口机制

### 2.1 视觉锚点

停泊坞是一个**从界面正中央底部向上弹出的浮动卡片**，其行为遵循：

```
┌─────────────────────────────────────────────────────────┐
│                    主界面（任意项目）                      │
│                                                         │
│   [文本栏 / 流程图栏]          [文本栏 / 流程图栏]         │
│                                                         │
│                                                         │
│                   ▲  停泊坞展开方向                       │
│   ┌───────────────────────────────┐  ← 宽度约 480-560px  │
│   │         停泊坞卡片             │  ← 与底边距离 16px    │
│   │   [任务资源池 / 专注控制台]    │                      │
│   └───────────────────────────────┘                     │
│              底部导航栏 / safe-area                       │
└─────────────────────────────────────────────────────────┘
```

> **关键约束**：停泊坞不是嵌入到某个项目中的，而是浮动在整个应用层。**切换项目不会触发停泊坞收回**。

### 2.2 触发展开

| 触发方式 | 说明 |
|---------|------|
| 点击触发胶囊 | 底部 TriggerCapsule（同 `parking-dock-modular-design.md` M7） |
| 拖拽任务块悬停 | 拖拽任意任务块悬停在触发胶囊上方 500ms 自动展开 |
| 快捷键 `Alt+Shift+D` | 快速展开/收起停泊坞（专注模式专用，与 Parking Dock `Alt+Shift+P` 区分） |

### 2.3 可拖入的内容类型

| 内容来源 | 类型 | 可拖入条件 |
|---------|------|----------|
| 文本栏的任务块 | Block/Task Item | `isCompleted === false`（仅未完成） |
| 流程图栏的节点 | GoJS Node（任务类型） | `isCompleted === false` |
| 黑匣子条目 | BlackBoxEntry | `isCompleted === false` |
| 停泊坞内新建 | 就地创建 | 无前置条件 |

> **硬约束**：已完成任务块被拖入时，停泊坞边缘变红，松手后弹出提示"仅可停泊未完成任务"并拒绝入坞。

### 2.4 首次主任务选择规则（补全 G-01，v3.0 修订）

```
首次拖入批次（session 内第一次拖入操作，即专注模式开启前拖入停泊坞）

  用户一次性拖入 N 个任务块（N ≥ 1）：
  ├─ N = 1：唯一任务直接成为主任务（绝对 C 位）
  └─ N > 1：
       ├─ v3.0 修订：算法选择拖入批次中的「第一个」任务为主任务
       │   └─ 「第一个」定义：批量拖入时按物理拖入顺序（draggedInAt 时间戳最小者）
       │   └─ 若完全同时（时间戳差 < 50ms）：按源任务在列表中的自然顺序
       └─ 用户可在主任务确认提示（3s Toast）内点选任意任务覆盖系统选择

后续批次（专注模式开启后，关闭虚化状态下再次拖入）：
  └─ v3.0 修订：不执行主任务选取，新拖入任务全部进入备选区域
  └─ 用户自行决定是否通过点选切换主任务（§2.8 手动点选机制）

判定逻辑：
  · 专注模式开启前拖入停泊坞 → 首次批次 → hasFirstBatchSelected = true
  · 专注模式开启后关闭虚化拖入 → 二次批次 → 不触发主任务选取
  · 注意：虚化开启时禁止拖拽操作（§9.3 v3.0 修订）
```

**Session 定义**：从用户首次点击"专注模式"按钮（灯泡 SVG，§2.7 v3.0）开始，到用户主动点击关闭专注模式并确认结束为止（§18.18 v3.0：跨会话持久保留）。

### 2.5 跨项目持久化（补全跨项目需求）

```typescript

 * FocusSessionState — 专注控制台全局状态

 */

**任务卡区域分发算法（补全 G-21）**：

```typescript
/**
 * 点击「专注模式」时，对停泊坞内所有任务的区域分配
 * v3.0：使用三维推荐阵列替代 computeRadarScore
 */
function assignZonesOnFocusStart(
  tasks: FocusTaskSlot[],
  mainTaskSlotId: string
): ZoneAssignment[] {
  const main  = tasks.find(t => t.slotId === mainTaskSlotId)!;
  const rest  = tasks.filter(t => t.slotId !== mainTaskSlotId);

  // 三维推荐阵列：每组最多 2 个候选（§4.2.4）
  const recommendation = computeThreeDimensionalRecommendation(main, rest);
  const comboSelectIds = new Set(
    recommendation.flatMap(g => g.candidates.map(c => c.slotId))
  );

  // 前 3 名进主控台卡片叠（第 2、3、4 层候命）
  const commandStack   = rest.filter(t => !comboSelectIds.has(t.slotId)).slice(0, 3);
  // 三维推荐命中 → 组合选择区域
  const comboTasks     = rest.filter(t => comboSelectIds.has(t.slotId));
  // 其余 → 备选区域
  const backupTasks    = rest.filter(t =>
    !comboSelectIds.has(t.slotId) && !commandStack.includes(t)
  );

  return [
    { zone: 'command',      tasks: [main, ...commandStack] },
    { zone: 'combo-select', tasks: comboTasks              },
    { zone: 'backup',       tasks: backupTasks             },
  ];
}
// 超出 22 个：按 §18.9 规则溢出队列
```

    · 同项目内部按 draggedInAt 升序
    · 不同项目间按各组最早 draggedInAt 升序
    · 注意：停泊坞排序不影响备选区域的空间分布算法

  主任务固定第一位：
    · 显示 indigo「◉ 主任务」标签
    · 不可被拖拽排序移走（位置锁定）

  主任务与专注状态解耦（v3.0 新增，补全 G-26 / G-38）：
    · isMaster 和 focusStatus 是独立属性
    · 主任务（isMaster=true）≠ 正在专注的任务（focusStatus='focusing'）
    · C 位卡片切换不改变 isMaster 标记
    · 只有以下操作改变 isMaster：
      ① 用户手动点选新的主任务（见下方操作路径）
      ② 当前主任务 completed → 自动清除 isMaster，第二位任务继承
      ③ 当前主任务被移出停泊坞 → 同②

  手动点选主任务（补全 G-26）：
    操作路径：
      桌面端：右键任意任务卡 → 浮动菜单 [◉ 设为主任务] [✕ 移出] [✏ 编辑属性]
      移动端：长按 500ms → 弹出同样的底部操作表
    点选逻辑：
      · 若 hasFirstBatchSelected = false → 设为主任务 + hasFirstBatchSelected = true（跳过 Toast）
      · 若 hasFirstBatchSelected = true  → 直接切换主任务标签（无 Toast 确认，瞬时生效）
    视觉反馈：
      · 旧主任务标签 200ms fade out，新主任务标签 200ms fade in
      · 卡片排序重排：新主任务动画平滑移至顶部（200ms ease-out reorder）

  用户手动拖拽重排（非主任务卡）：
    · 长按 500ms → 进入拖拽模式，竖向重排
    · 重排只影响面板内显示顺序，不影响三区算法评分（score 不随显示顺序变化）
    · 专注中面板展开时：禁止内部重排（防止误操作），拖拽手势改为仅支持「拖出移除」

  跨会话主任务继承（补全 G-37，v3.0 修订）：
    · hasFirstBatchSelected 跨会话持久化（v3.0 覆盖 v2.1 的"每次重置"逻辑）
    · 退出专注保留停泊坞（§18.18 默认行为），FocusSessionState 中 isMaster=true 的任务
      即为上次主任务
    · 下次再进入专注时：
      - 若上次主任务仍在停泊坞且 focusStatus ≠ 'completed'：直接沿用为主任务（3s Toast 可更改）
      - 若已完成或已被移除：重新走 §2.4 首次选择算法
    · 应用关闭/崩溃重启：自动恢复停泊坞状态和主任务标记（IndexedDB 持久化）

```typescript
/**
 * FocusTaskSlot — 停泊坞专注控制台中的任务槽
 * 
 * 基础任务数据来自原始 Task 实体（通过 taskId 引用）。
 * FocusTaskSlot 仅持有专注模式独有的扩展属性。
 */
interface FocusTaskSlot {
  slotId: string;           // 槽 ID，crypto.randomUUID()
  taskId: string | null;    // 关联原始任务 ID（null = 就地创建的临时任务）
  
  // ─── 专注属性 ───
  estimatedMinutes: number | null;  // a. 总预计时间（分钟，含等待时间，null = 未设置）
                                    //    语义：estimatedMinutes = 有效执行时间 + 等待时间
                                    //    示例：API 调用总计 45min，其中 20min 等待响应
                                    //    → estimatedMinutes = 45，waitMinutes = 20
                                    //    → effectiveExecMinutes = 25（纯粹需要专注投入的时间）
  waitMinutes: number | null;       // c. 等待时间（分钟，≤ estimatedMinutes，null = 无等待）
                                    //    约束：0 < waitMinutes ≤ estimatedMinutes（UI 层校验）
  cognitiveLoad: 'high' | 'low';    // b. 认知负荷（v3.0：默认 'low'，不再允许 null）
  
  // ─── 专注状态 ───
  focusStatus: FocusTaskStatus;     // 当前在控台中的状态
  zone: 'command' | 'combo-select' | 'backup';  // v3.0：组合选择区域 + 备选区域（原 radar/periphery）
  zoneIndex: number;                // 区域内排列位置（orbit index）
  
  // ─── 主任务标记（v3.0 新增，补全 G-38）───
  isMaster: boolean;                // 是否为主任务（true = 主任务，C 位专注对象）
                                    // 语义：isMaster 与 focusStatus='focusing' 解耦
                                    //   · isMaster=true + focusing：当前正在专注的主任务
                                    //   · isMaster=true + suspend-waiting：主任务挂起等待
                                    //   · isMaster=false + focusing：非主任务但正在专注（碎片阶段）
                                    // 持久化：跨会话保留（直到用户主动更换或任务完成）
  
  // ─── 等待计时 ───
  waitStartedAt: number | null;     // 等待开始时间戳（null = 未在等待）
  waitEndAt: number | null;         // 等待结束时间戳
  
  // ─── 来源信息 ───
  sourceProjectId: string | null;   // 来源项目 ID
  sourceBlockType: 'text' | 'flow' | null;  // 来源板块类型
  draggedInAt: number;              // 拖入时间戳
  isFirstBatch: boolean;            // 是否属于首次拖入批次
  
  // ─── 就地创建专属 ───
  inlineTitle: string | null;       // 就地创建时的标题（原始任务无此字段）
  inlineDetail: string | null;      // 就地创建时的详情
}

type FocusTaskStatus =
  | 'pending'              // 待启动：在组合选择区域或备选区域等待
  | 'focusing'             // 专注中：位于绝对 C 位
  | 'suspend-waiting'      // 挂起等待：计时倒数中，位于卡片叠后方
  | 'wait-ended'           // 等待结束：铃声提示，边缘发光，等待用户操作
  | 'completing'           // 完成动画中（过渡状态）
  | 'completed';           // 已完成，已飞出控制台
```

#### 4.2.1 轨道分布规则（v3.0 修订：线→面自适应，补全 G-06）

```typescript
/**
 * v3.0 三维推荐阵列算法
 * 替代原 computeRadarScore 单分数排序
 * 每组加入时间约束 T_est ≤ W_main（v3.0 闭环修订）
 */
interface RecommendationGroup {
  type: 'homologous-advancement' | 'cognitive-downgrade' | 'asynchronous-boot';
  candidates: FocusTaskSlot[];
  isOversized?: boolean;  // 大任务回退标记（空集处理时为 true）
}

function computeThreeDimensionalRecommendation(
  mainTask: FocusTaskSlot,
  pendingTasks: FocusTaskSlot[],
  remainingWaitMin: number
): RecommendationGroup[] {
  const assigned = new Set<string>(); // 防止同一任务出现在多组（正常模式）

  // ① 同源推进（Homologous Advancement）：P = True AND T_est ≤ W_main
  const homologousAdvancement = pendingTasks
    .filter(t => t.sourceProjectId === mainTask.sourceProjectId
             && (t.estimatedMinutes ?? Infinity) <= remainingWaitMin
             && !assigned.has(t.slotId))
    .sort((a, b) => effectiveExecMin(a) - effectiveExecMin(b))
    .slice(0, 2);
  homologousAdvancement.forEach(t => assigned.add(t.slotId));

  // ② 认知降级（Cognitive Downgrade）：L = Low AND T_est ≤ W_main，优先 P = False
  const cognitiveDowngrade = pendingTasks
    .filter(t => t.cognitiveLoad === 'low'
             && (t.estimatedMinutes ?? Infinity) <= remainingWaitMin
             && !assigned.has(t.slotId))
    .sort((a, b) => {
      // 优先推荐不同项目（P = False），彻底转移注意力
      const pA = a.sourceProjectId === mainTask.sourceProjectId ? 1 : 0;
      const pB = b.sourceProjectId === mainTask.sourceProjectId ? 1 : 0;
      if (pA !== pB) return pA - pB; // P=False 排前面
      return Math.abs(effectiveExecMin(a) - remainingWaitMin) -
             Math.abs(effectiveExecMin(b) - remainingWaitMin);
    })
    .slice(0, 2);
  cognitiveDowngrade.forEach(t => assigned.add(t.slotId));

  // ③ 异步并发（Asynchronous Boot）：T_wait > 0 AND effectiveExecMin ≤ W_main
  //    （时间约束为启动时间而非总时间，因为等待部分可与主任务重叠）
  const asynchronousBoot = pendingTasks
    .filter(t => (t.waitMinutes ?? 0) > 0
             && effectiveExecMin(t) <= remainingWaitMin
             && !assigned.has(t.slotId))
    .sort((a, b) => (b.waitMinutes ?? 0) - (a.waitMinutes ?? 0)
                  || effectiveExecMin(a) - effectiveExecMin(b))
    .slice(0, 2);

  // ═══ 空集检测与大任务回退（补全 G-13 边界）═══
  const allEmpty = homologousAdvancement.length === 0
                && cognitiveDowngrade.length === 0
                && asynchronousBoot.length === 0;

  if (allEmpty && pendingTasks.length > 0) {
    if (remainingWaitMin >= 30) {
      // W_main ≥ 30min → 放宽时间约束，选出"大任务"推荐
      // 大任务模式下属性允许重叠（同一任务可出现在多组中）
      return computeOversizedFallback(mainTask, pendingTasks);
    }
    // W_main < 30min → 不弹出推荐，直接返回空集（调用方进入留白期）
    return [
      { type: 'homologous-advancement', candidates: [] },
      { type: 'cognitive-downgrade',    candidates: [] },
      { type: 'asynchronous-boot',      candidates: [] },
    ];
  }

  return [
    { type: 'homologous-advancement', candidates: homologousAdvancement },
    { type: 'cognitive-downgrade',    candidates: cognitiveDowngrade },
    { type: 'asynchronous-boot',      candidates: asynchronousBoot },
  ];
}

/**
 * 大任务回退推荐（空集处理 W_main ≥ 30min）
 * 放宽时间约束，属性允许重叠
 */
function computeOversizedFallback(
  mainTask: FocusTaskSlot,
  pendingTasks: FocusTaskSlot[]
): RecommendationGroup[] {
  // 同源推进：同项目，按 T_est 升序取最短的 1 个
  const sameProject = pendingTasks
    .filter(t => t.sourceProjectId === mainTask.sourceProjectId)
    .sort((a, b) => (a.estimatedMinutes ?? Infinity) - (b.estimatedMinutes ?? Infinity))
    .slice(0, 1);

  // 认知降级：低负荷，按 T_est 升序取最短的 1 个（允许重叠）
  const lowLoad = pendingTasks
    .filter(t => t.cognitiveLoad === 'low')
    .sort((a, b) => (a.estimatedMinutes ?? Infinity) - (b.estimatedMinutes ?? Infinity))
    .slice(0, 1);

  // 异步并发：有等待时间，按 effectiveExecMin 升序取 1 个（允许重叠）
  const hasWait = pendingTasks
    .filter(t => (t.waitMinutes ?? 0) > 0)
    .sort((a, b) => effectiveExecMin(a) - effectiveExecMin(b))
    .slice(0, 1);

  return [
    { type: 'homologous-advancement', candidates: sameProject, isOversized: true },
    { type: 'cognitive-downgrade',    candidates: lowLoad,     isOversized: true },
    { type: 'asynchronous-boot',      candidates: hasWait,     isOversized: true },
  ];
}

function effectiveExecMin(t: FocusTaskSlot): number {
  return (t.estimatedMinutes ?? 0) - (t.waitMinutes ?? 0);
}
```

### 6.2 状态 "d: 专注中" 精确定义

- 任意时刻**有且仅有一个**任务处于「专注中」状态
- 专注中任务 = C 位卡片（最前方、100% 不透明）
- StatusHUD 显示该任务的「专注中」脉冲圆
- 该任务若有等待时间且到期 → 转为「等待结束」

### 6.3 状态 "e: 挂起等待" 精确定义

- 任务进入「挂起等待」后：
  1. 移出 C 位，插入卡片叠最后方（降至 25% 透明度）
  2. StatusHUD 为该任务显示环形倒计时
  3. 算法自动从组合选择区域/备选区域选出新任务晋升为新的 C 位
  4. 多个任务同时挂起时：**仅第一个进入挂起的任务触发算法推荐**（后续任务等前者恢复再做判断）

## 七、等待时间调度算法

### 7.1 算法触发时机

| 触发事件 | 算法目标 |
|---------|---------|
| 任务进入「挂起等待」 | 从组合选择区域/备选区域选出最优穿插任务 |
| 穿插任务完成 | 判断剩余穿插时间是否还能再做一个任务 |
| 主任务等待结束 | 柔和提示，等待用户确认恢复 |
| 所有任务均挂起 | 进入碎片阶段，展示日常任务 |

### 7.2 穿插任务选择算法

```typescript
/**
 * 等待时间穿插任务选择算法
 * 
 * @param waitingTask - 刚进入挂起等待的任务
 * @param candidates - 当前组合选择区域 + 备选区域中 status = 'pending' 的任务
 * @returns 推荐任务列表（前 3 个，供用户从中选择）
 */
function selectInterludeTask(
  waitingTask: FocusTaskSlot,
  candidates: FocusTaskSlot[]
): FocusTaskSlot[] {
  const remainingWaitMs = computeRemainingWaitMs(waitingTask);
  
  return candidates
    .map(task => ({
      task,
      score: computeInterludeScore(task, remainingWaitMs),
    }))
    .filter(({ score }) => score > 0)
    .sort((a, b) => b.score - a.score)
    .slice(0, 3)
    .map(({ task }) => task);
}

function computeInterludeScore(task: FocusTaskSlot, remainingWaitMs: number): number {
  const remainingMin = remainingWaitMs / 60_000;
  
  // 修正 G-22：使用「有效执行时间」= estimatedMinutes − waitMinutes
  // 避免将任务自身的等待时间计入对比，造成系统性高估
  if (!task.estimatedMinutes) return 5; // 无时间估算：低分但仍可推荐
  
  const effectiveExecMin = task.estimatedMinutes - (task.waitMinutes ?? 0);
  
  // 时间严重超出（有效执行时间 > 剩余等待时间的 150%）→ 极低分
  if (effectiveExecMin > remainingMin * 1.5) return 2;
  
  // 时间远低于剩余等待时间（有效执行时间 < 20%）→ 轻微扣分（任务太短，留白感强）
  const timeFitScore = effectiveExecMin > remainingMin * 0.2 ? 25 : 10;

  // 低认知负荷优先（+20）— 等待间隙宜选轻松任务
  const loadScore = task.cognitiveLoad === 'low' ? 20 : 0;  // v3.0：不再允许 null
  
  // 有等待时间（可与主任务等待叠加执行，适合嵌套，+15）
  const waitBonusScore = task.waitMinutes ? 15 : 0;
  
  // 组合选择区域优先于备选区域（+10 vs +5）
  const zoneScore = task.zone === 'combo-select' ? 10 : 5;
  
  return timeFitScore + loadScore + waitBonusScore + zoneScore;
}
```

### 7.3 算法边界条件（补全 G-03）

```
情况一：等待时间到期，但 B 任务仍在进行中
  → 系统仅发出柔和提示（chime + StatusHUD 光晕）
  → 不强制中断 B 任务
  → B 任务的 StatusHUD 圆圈由「专注中」变为「专注中 (A 等待结束)」
  → 用户选择：
      [a] 继续 B，稍后手动切换 A（拖拽 A 从卡片叠后方到前方）
      [b] 点击 StatusHUD 中的 A 状态条 → A 执行「抽卡插入」动画（见 §8.5）
  → 系统不会 10s 内自动二次提醒，避免心流中断

情况二：等待时间到期，B 任务已完成
  → B 向上飞出
  → A 自动从卡片叠后方推进到 C 位（参见 §8.6）
  → 无需用户干预

情况三：设置了等待时间，但在等待期内主动取消等待（用户重新拖动 A 到 C 位）
  → 算法停止，当前穿插任务回退到组合选择区域
  → A 重新进入「专注中」状态
```

### 7.4 并发等待处理（补全 G-04）

```
当两个任务同时进入等待且等待时间接近：

  规则：按照「进入挂起等待的时间戳」排序，先挂起者先执行算法
  
  若时间戳差 < 1s（几乎同时）：
    → 按 cognitiveLoad 排序（高负荷先挂）
    → 再按 estimatedMinutes 排序（时间长的先挂）
    → 最终仍相同：按 slotId 字典序（保证确定性）
    
  只有第一个挂起的任务触发算法推荐，第二个任务挂起到卡片叠后方等待。
```

### 7.5 算法推荐不执行晋升，仅建议

> **重要设计哲学**：算法负责高亮推荐，**用户负责最终决策**，系统永不强制切换。

```
算法推荐流程：
  1. 计算推荐任务（1~3 个候选）
  2. 候选任务在组合选择区域/备选区域出现「虚线脉冲高亮」+ 磁力向主控台漂移倾向（位移 8px）
  3. StatusHUD 显示推荐标注（如「B 时间合适 ✓」）
  4. 用户决定：
     [a] 点击推荐任务 → 晋升为主任务（磁力吸附动画）
     [b] 无操作 3s → 最高分推荐任务自动晋升（给出 3s 进度条可取消）
     [c] 点击另一个非推荐任务 → 直接晋升该任务，推荐消失
```

### 7.6 碎片阶段（Fragment Phase）进入规则

```
进入条件：
  · 所有 status = 'focusing' 或 'suspend-waiting' 的任务
  · 无任何 status = 'pending' 的组合选择区域/备选区域任务可穿插

碎片阶段：
  → 控制台进入「留白期」视觉状态（整体透明度微降，界面轻微呼吸动画）
  → 日常任务槽自动展开（见 §10）
  → StatusHUD 切换为「留白期 - 全部任务等待中」
  
退出条件：
  · 等待时间最短的任务到期（StatusHUD 提示，用户点击恢复）
  · 用户拖入新任务（立即结束碎片阶段）
  
降级兜底（补全 G-07）：若等待时间 ≤ 2min 且无日常任务：
  · 不进入碎片阶段
  · 直接显示「距 [任务名] 恢复还有 X 分钟」的简洁倒计时提示
  · 静默等待，不展示任何新任务
```

### 7.7 穿插任务完成后的时间剩余判断（补全 G-14 / G-15）

> **场景**：主任务 A 已挂起等待，穿插任务 B 被算法选入并已完成，此时需判断 A 的剩余等待时间情况。

```
设变量：
  remainingWait = A.waitEndAt - Date.now()           // A 剩余等待毫秒数（毫秒）
  remainingMin  = remainingWait / 60_000              // 换算为分钟
  C = 算法此前预选的下一个候选任务（排在 B 后面）
  C.effectiveExecMin = C.estimatedMinutes - (C.waitMinutes ?? 0)  // 补全 G-22

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  情况 0：remainingMin ≤ 2
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    → 直接进入「留白期」（不再调度新任务）
    → 显示简洁倒计时条："[任务 A] 将在 Xm Xs 后恢复"
    → C 保持在组合选择区域/备选区域，不移动，不高亮

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  情况 0.5（补全 G-25）：有剩余时间，但太紧/完成 C 后 A 几乎立即到期
  条件：remainingMin > 2  且  C.effectiveExecMin > remainingMin × 0.75
        （即 C 要消耗剩余等待时间的 75% 以上，几乎没有缓冲）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    → 进入「紧张留白期」（轻量版，区别于情况 0 的完全留白）：
    
      [a] 界面显示「准备迎接 [任务 A]」轻量提示条（无大号圆环）
          文案：「A 即将恢复（约 Xmin），C 仅有 Y% 时间余量」
      
      [b] C 以淡黄色虚线边框展示（非强推荐，表示「时间紧，可尝试但谨慎」）
          透明度轻微升高至 0.80（比正常待启动更亮，表示「被考虑中」）
      
      [c] 用户决定（5s 超时进入完全留白期，有进度条可取消）：
          · 点击 C → 强制晋升（用户自行承担完不成的风险），算法不阻止
          · 5s 不操作 → 进入完全留白期（与情况 0 相同），C 恢复正常透明度
      
      [d] 若 C.effectiveExecMin ≤ remainingMin × 0.40（任务足够轻松）：
          升回情况 1 处理分支（时间余量充足不需要警告）

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  情况 1：C.effectiveExecMin × 0.75 ≤ remainingMin ≤ C.effectiveExecMin × 1.5
         （C 的有效执行时间与剩余等待时间基本匹配，补全 G-22）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    → C 保持预选资格，直接执行磁力吸附晋升（§8.4 动画序列）
    → B 执行飞出动画（§8.2）后，C 从组合选择区域/备选区域飘入主控台
    → StatusHUD 更新：C 变为"专注中"，A 保持"挂起等待"环形倒计时
    → 若 C 也有等待时间：进入 §7.9 多层等待嵌套规则

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  情况 2：C 时间严重超出或远小于 A 剩余时间
         （超出定义：C.effectiveExecMin > remainingMin × 1.5）
         （偏小定义：C.effectiveExecMin < remainingMin × 0.2）
         （均使用有效执行时间，补全 G-22）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    → B 飞出后，触发重新算法推荐：
    
      [a] C 透明度逐渐降低（从 65% 降至 40%，800ms 渐变），
          同时位置向外围方向轻微漂移（8px），视觉上"退出推荐"
      
      [b] 算法重新从组合选择区域/备选区域计算最优候选 D（computeInterludeScore()）
      
      [c] D 出现虚线高亮脉冲边框，并显示推荐标注
          （如"D 时间合适 ✓ 剩余 18min，D 有效执行 15min"）
      
      [d] C 和 D 同时保持可见，状态标注：
          C = "📊 偏差较大，仍可选"（橙色标注）
          D = "✓ 算法推荐"（indigo 标注）
      
      [e] 用户决定：
          · 点击 D → 磁力吸附晋升 D，C 退回正常外围/雷达透明度
          · 点击 C → 直接晋升 C（用户覆盖算法），D 高亮消失
          · 3s 不操作 → D 自动晋升（可取消进度条，见 §18.4）
      
      [f] 已完成的 B 任务从停泊坞彻底消失（isCompleted = true，飞出后 500ms 移除槽位）

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  A 挂起等待结束事件（在以上任何情况下均可能发生）
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    → 系统发出柔和提示音（见 §5.3 chime 规格）
    → A 在卡片叠后方出现边缘呼吸发光（indigo 光晕）
    → StatusHUD 中 A 的状态从「挂起等待」变为「等待结束」+ 圆环进入呼吸发光
    → 不干扰当前 C 位任务的心流，不做强制界面切换
    → 用户选择：
        [a] 点击 StatusHUD 中 A 的条目 → 执行「抽卡插入」动画（§8.5），A 接管 C 位
        [b] 等待当前 C 位任务完成后 → A 自动从后方推进（§8.6）
        [c] 继续忽略（A 保持「等待结束」状态悬挂在卡片叠后方，呼吸光晕持续）
```

### 7.8 碎片处理阶段完整流程（补全 G-16 / G-17 / G-34 / G-40，v3.0 修订：四级防御体系）

> **进入条件**：主控台和组合选择区域、备选区域中，所有 `FocusTaskSlot` 均处于 `suspend-waiting`（含 `wait-ended`），且无任何 `pending` 任务可穿插。
>
> **算法状态声明（补全 G-40）**：进入碎片阶段时，FocusSchedulerService 进入「暂停」状态——不响应任何任务状态变化触发的新推荐计算。碎片阶段退出时（正常退出 Step 6 或用户拖入 Step 5），调度算法完全重置（清除上次推荐缓存），重新对全部 pending 任务执行三维推荐阵列计算。

**v3.0 四级防御体系（补全 G-16）**：

```
碎片阶段不再是简单的「等待 + 做日常任务」，而是一个结构化的四级防御系统：

  ═══════════════════════════════════════════════════════
  Level 1：被动触发（Passive Trigger）— 等待时间 ≤ 5min
  ═══════════════════════════════════════════════════════
    · 不进入完整碎片阶段
    · 仅显示倒计时提示卡（Step 3）
    · 不展示日常任务、不推荐碎片事件
    · 用户可自由操作应用（虚化层不加深）

  ═══════════════════════════════════════════════════════
  Level 2：主动触发 / 倦怠检测（Active Trigger / Burnout）— 等待 > 5min
  ═══════════════════════════════════════════════════════
    · 进入标准碎片阶段（Step 1~6 流程）
    · 倦怠检测算法（补全 NG-16b）：
      ┌─────────────────────────────────────────────────┐
      │ 条件：过去 2h 内连续完成 ≥ 3 个 cognitiveLoad   │
      │       = 'high' 的任务                           │
      │ 数据源：FocusSessionState.highLoadCounter       │
      │   · 每次完成 high 任务：                        │
      │     - 若 windowStartAt 距现在 > 2h → 重置       │
      │       { count: 1, windowStartAt: Date.now() }   │
      │     - 否则 → count++                            │
      │   · count ≥ 3 → 触发强制碎片阶段                │
      │     即使当前无等待任务也强制进入 Level 2         │
      │     burnoutTriggeredAt = Date.now()              │
      │   · 触发后 count 重置为 0                       │
      │   · 计数器跨会话持久化（IndexedDB 序列化）      │
      │   · 非专注模式完成高负荷任务也计入（G-18）      │
      └─────────────────────────────────────────────────┘
    · 展示碎片事件三类（按优先级顺序推荐）：
      ┌──────────────────────────────────────────────────┐
      │ a. 身体跨界（Physical Crossover）——  优先级最高    │
      │    · 站起来倒杯水、伸展、洗手、看窗外             │
      │    · 系统预置 5~8 个常见条目，用户可自定义          │
      │    · 时长建议：2~5min                              │
      ├──────────────────────────────────────────────────┤
      │ b. 数字清洁工（Digital Janitor）——  次优先级       │
      │    · 清理通知、整理文件、归档邮件                   │
      │    · 从用户的「日常任务槽」（§10）拉取              │
      │    · 时长建议：3~8min                              │
      ├──────────────────────────────────────────────────┤
      │ c. 微进度（Micro-Progress）——  最低优先级          │
      │    · 当前项目的极小粒度子任务                       │
      │    · 从备选区域中 effectiveExecMin ≤ 5 的任务拉取   │
      │    · 时长建议：≤ 5min                               │
      └──────────────────────────────────────────────────┘

  ═══════════════════════════════════════════════════════
  Level 3：分类优先级（Classification Priority）— 多碎片排队
  ═══════════════════════════════════════════════════════
    · 碎片事件按 a > b > c 顺序展示
    · 做完一个碎片事件后：
      ★ 不连发！直接进入 Level 4（Zen Mode）
      → 避免碎片事件无限连续消耗用户精力
    · 用户可手动点击「再来一个」跳过 Zen Mode 继续做碎片

  ═══════════════════════════════════════════════════════
  Level 4：消耗/重置 + Zen Mode — 碎片完成后
  ═══════════════════════════════════════════════════════
    · 碎片事件完成 → 自动进入 Zen Mode（禅模式）
    · Zen Mode 视觉：
      - 界面完全留白，仅保留呼吸动画圆（中心 60px 脉冲圆）
      - 背景完全模糊（blur 8px）
      - 文案：「放空一会儿...」
      - 倒计时提示仍在右上角 StatusHUD 小号显示
    · Zen Mode 退出条件：
      - 等待任务到期 → 自动退出，恢复正常
      - 用户点击屏幕任意位置 → 退出 Zen Mode
      - 用户点击「再来一个碎片」→ 回到 Level 2
    · Zen Mode 持续时间不设上限（可以一直放空到任务恢复）
```

### 7.9 多层等待嵌套推进规则（补全 G-16）

> **场景**：A 挂起启动 B → B 也设了等待并挂起 → C 从组合选择区域推进 → C 也设了等待。

```
规则：只对第一层挂起任务（最先进入 suspend-waiting 的任务）执行算法推荐。

  具体推进栈（示例）：

  初始：A 专注中
  ↓ A 触发挂起（30min 等待）
    算法选 B 进入主控台，A 下沉到卡片叠后方（第 2 或 3 层）
    StatusHUD：A = 挂起等待, B = 专注中

  ↓ B 也触发挂起（15min 等待）
    规则：第二个挂起任务（B）不会再触发算法推荐
    B 下沉到卡片叠后方，位于 A 的前面（B 更晚挂起，排在 A 前）
    算法不自动选 C，而是等待用户手动操作
    
    ── 但有例外：若 B 的等待时间 ≤ A 剩余时间的 50%（B 很快恢复），
                则不进行 C 的选择；
                若 B 的等待时间 > A 剩余时间（B 会比 A 晚结束），
                且当前 pending 任务中有符合 B 等待时间的 C：
                → 系统给出「可选」提示（非强制），用户确认后 C 进入主控台

  ↓ 若 B 等待到期（先于 A 结束）
    B 从卡片叠后方推进到 C 位（§8.6 动画），StatusHUD 更新
    A 继续等待，保持挂起

  ↓ A 等待到期（B 仍在专注或已完成）
    若 B 仍在进行：仅 StatusHUD 呼吸光晕提示（§7.3 情况一逻辑）
    若 B 已完成：A 自动推进到 C 位（§8.6 动画）

  ↓ 若 C 进入主控台且也触发挂起
    此时 A,B,C 均挂起 → 触发碎片处理阶段（§7.8）
    所有等待任务中时间最短的控制倒计时提示卡

多层嵌套上限：
  · 最多允许 2 层同时挂起（A + B）时系统仍正常调度
  · 3 层及以上同时挂起 → 自动进入碎片阶段，不再尝试调度
  · StatusHUD 最多展示 3 个挂起任务的倒计时圆环
    超过 3 个挂起任务：StatusHUD 折叠显示"+ N 个等待中"
```

## 九、虚化层（Blur Layer）——与专注模式解耦（v3.0 修订，补全 G-19）

> v3.0 核心修订：虚化（Blur）和专注（Focus）是两个独立开关。
> 虚化 = 视觉遮罩，降低背景干扰。专注 = 任务调度系统。
> 用户可以「只虚化不专注」或「只专注不虚化」。

### 9.0 双开关独立控制（v3.0 新增）

```
两个独立按钮：

  ① 虚化按钮（Blur Toggle）：
     · 位置：原「专注模式」按钮位置（停泊坞面板标题旁）
     · 图标：EyeOff 图标 + 文字「虚化」
     · 作用：仅控制背景虚化遮罩层，不影响任务调度
     · 快捷键：Alt+Shift+F（保留原快捷键）
     · 状态枚举：
       - 关闭：按钮灰色，主内容区清晰
       - 开启：按钮紫色，主内容区虚化

  ② 专注按钮（Focus Toggle）：
     · 位置：项目栏（Project Bar）中，录音按钮上方
     · 图标：灯泡 SVG（Lightbulb）
     · 作用：启动/关闭专注会话和三区调度系统
     · 快捷键：Alt+Shift+L
     · 状态枚举：
       - 停泊坞无任务：按钮置灰
       - 未进入专注：按钮激活（indigo 色）
       - 专注进行中：灯泡亮起（amber 色），点击→触发 §18.18 退出流程

组合状态矩阵：
  ┌─────────────────┬──────────────────────┬──────────────────────┐
  │                  │ 虚化 OFF              │ 虚化 ON              │
  ├─────────────────┼──────────────────────┼──────────────────────┤
  │ 专注 OFF        │ 正常模式              │ 仅虚化（纯阅读模式） │
  │                  │ 无遮罩，无三区        │ 背景虚化，无三区      │
  ├─────────────────┼──────────────────────┼──────────────────────┤

  │                  │ 无遮罩，三区可见      │ 背景虚化 + 三区可见   │

  └─────────────────┴──────────────────────┴──────────────────────┘

### 9.1 虚化层设计（v3.0 修订）

```
触发：用户点击「虚化」按钮（独立于专注按钮）

  开启虚化：
    · 应用主内容区（文本栏、流程图栏、侧边栏等）
    · 使用：backdrop-filter: blur(6px) + overlay: rgba(10,10,20,0.45)
    · 过渡：500ms ease-in
    · 专注控制台区域不受虚化影响（z-index 高于遮罩）
    · 停泊坞和 StatusHUD 不受影响（z-index 高于遮罩）

  关闭虚化：
    · 再次点击「虚化」按钮 或 按 Escape
    · overlay 消失：500ms ease-out
    · 主内容恢复清晰
    · 若专注模式仍在运行 → 三区控制台保持显示（透明专注模式）
```

### 9.2 虚化层层次结构

```
z-index 层次（v3.0 更新，从低到高）：
  100  → 主内容区（文本栏、流程图栏）
  200  → 侧边栏
  500  → 上下文弹出菜单
  800  → 虚化遮罩（仅虚化按钮开启时存在）
  900  → 停泊坞卡片面板
  950  → 专注控制台三区（主控台、组合选择区域、备选区域）
  1000 → StatusHUD 悬浮挂件
  1050 → 视觉连线（来自 parking-dock-modular-design.md M19.15）
  1100 → Toast / NoticeBar
  9999 → 模态框（如大门 Gate）
```

### 9.3 虚化中拖入新任务（补全 G-08，v3.0 修订）

```
场景 A：虚化 ON + 专注 ON → 从虚化的文本栏拖出任务块

处理方式：保持虚化，但给予清晰的拖入反馈
  · 拖拽物（任务块）本身不受虚化影响（z-index 超过遮罩）
  · 停泊坞触发胶囊高亮（明亮轮廓），明确引导拖入目标
  · 任务进入停泊坞后，短暂（1s）显示「已加入停泊坞」行内提示，然后消失
  · 不会因为拖入操作而关闭或中断虚化

场景 B（v3.0 新增）：虚化 ON + 专注 OFF → 虚化层禁止拖拽
  · 虚化模式下未开启专注时，虚化层拦截所有拖拽事件
  · 拖拽到虚化层上方 → 显示 Toast："请先开启专注模式"
  · 设计意图：虚化是为了减少视觉干扰，不应在此模式下操作内容

注意：「确认拖入了已完成任务」的红色拒绝提示不被虚化遮罩阻挡
```

### 10.2 碎片阶段展示逻辑（v3.0 修订：静默消失动画，补全 G-35）

```
日常任务展示规则（补全 §7.6 降级兜底 G-07）：

  条件：
    1. 处于碎片阶段（所有任务挂起等待）
    2. 当日该日常任务 showCount < maxTimesPerDay
    3. 任务 isEnabled === true

  展示方式：
    · 日常任务以「坠入动画」从界面上方飞入，停在主控台 C 位区域
    · 与挂起任务共存（挂起任务自动缩小至第 2~3 层）
    · 日常任务卡的视觉风格略有不同：淡蓝色左侧色条 + 「日常」标签

  完成日常任务（v3.0 修订：静默消失）：
    · 点击「✓ 完成」→ 静默淡出（不飞出）+ showCount + 1 → 进入 Zen Mode（§7.8 Level 4）
    · 若 showCount 达到 maxTimesPerDay → 当日不再显示

  完成动画（v3.0 修订，补全 G-35）：
    · 碎片任务/日常任务完成：静默淡出（不使用飞出动画）
    · opacity 1 → 0，400ms ease-out
    · 无位移、无旋转、无缩放——原地消失
    · 设计意图：碎片任务是轻量级的，飞出动画过于张扬，静默消失更符合"放松"语境

  坠入动画（入场动画不变）：
    translateY(-100vh) opacity(0) → translateY(0) opacity(1)
    持续 400ms，cubic-bezier(0.16, 1, 0.3, 1)
```

### 10.3 降级兜底（补全 G-07）

```
若碎片阶段：
  · 无已启用的日常任务 AND
  · 所有等待任务的剩余时间 ≤ 2min

→ 不进入完整碎片阶段
→ 显示简洁「倒计时条」："[任务 X] 将在 Xm Xs 后恢复"

### 10.4 日常任务与专注会话的生命周期绑定（补全 G-18）

```
问题一：日常任务的"每日次数"是否与专注会话解耦？

  解耦规则：
  · 日常任务的完成计数（routine_completions 表）以「日历日期」为维度统计，
    与是否开启了专注会话无关
  · 用户在任何地方完成日常任务（停泊坞、日常任务管理面板），均会计入当天计数
  · 专注会话仅决定日常任务的「展示时机」（仅在碎片阶段自动弹出）
  · 用户随时可从日常任务管理面板手动触发完成，也会计入

  · 用户可在设置中自定义重置时间（5:00 = 夜猫模式）

  ③ 应用设置 → 专注模式 → 日常任务管理

  创建示例（预置模板，用户可修改）：
  · "服用维生素" – 每天 1 次
  · "喝水（250ml）" – 每天 4 次
  · "站起来活动 5 分钟" – 每天 4 次
  · [自定义标题] – 最多每天 N 次（N 可设为 1~20）

## 十一、跨项目持久化策略

### 11.1 停泊坞跨项目行为

| 操作 | 停泊坞行为 |
|------|----------|
| 用户切换到项目 B（当前在项目 A） | 停泊坞保持展开/收起状态，内容不清空 |

| 主动退出专注会话 | 清空 FocusSessionState，专注控制台消失，停泊坞回退为普通停泊模式 |

### 11.2 离线策略（补全 G-11）

```
停泊坞中的任务跨项目引用时的离线处理：

  · 所有 FocusTaskSlot 引用的 Task 数据优先从 IndexedDB 读取
  · 若任务所在项目在 IndexedDB 中尚未缓存 → 显示「加载中」占位，后台异步拉取
  · 离线状态下：
      - 在停泊坞内的所有操作（修改状态、创建就地任务、标记完成）
      - 均写入本地 IndexedDB + 标记 `pendingSync = true`
      - 网络恢复后通过 LWW 策略同步
  · 就地创建的任务（sourceProjectId = null）离线也可正常创建，归档时再同步
```

### 11.3 同步优先级

```
专注控制台数据的同步优先级：

  P0（立即同步）：任务完成状态（isCompleted = true）
  P1（3s 防抖同步）：认知负荷变更、等待时间设置
  P2（专注会话结束时批量同步）：FocusSessionState 快照
  P3（后台低优先级）：就地创建的临时任务归档

### 15.8 区域透明度丝滑过渡规则

> **核心约束**：相邻区域透明度差值不超过 30%，各卡片间有轻微相位错开的呼吸动画，让视觉层次「流动」而非「跳变」。

```
透明度层级（从前到后，丝滑递减）：

  主控台 C 位         opacity: 1.00   （白昼亮 100%）
       ↓ 差值 0.58
  主控台第 2 层       opacity: 0.42   （挂起时迅速降至此值）
       ↓ 差值 0.17
  主控台第 3 层       opacity: 0.25   （几乎成为影子）

  ── 区域间硬边界（不超过 0.28 差值）──

  组合选择区域：内层        opacity: 0.72 ~ 0.80  （与主控台关联最强的约 0.80）
  组合选择区域：外层        opacity: 0.62 ~ 0.70
       ↓ 差值最大 0.28
  备选区域              opacity: 0.38 ~ 0.50

  漂浮呼吸：每张卡片 opacity 在各自基础值 ±0.035 区间内缓慢呼吸
            周期 3.5s ~ 6s，每张卡片随机相位偏移（避免同步闪烁）

  背景虚化遮罩        rgba(10,12,20,0.40) — 永不参与透明度层级计算
```

```css
/* 呼吸透明度动画（各卡片设置不同 animation-delay 实现异相） */
@keyframes card-breathe {
  0%, 100% { opacity: var(--card-opacity-base); }
  50%       { opacity: calc(var(--card-opacity-base) + 0.035); }
}
```

## 十八、v1.1 深度逻辑完善（18 项补充修正）

> 本节是对 v1.0 的批判性二次审查，补充了首次审查中遗漏的边界条件、时序冲突、算法盲区和 UX 一致性问题。

### 18.1 多任务同时完成的竞态处理

```
场景：用户快速连续点击两个任务的"完成"按钮（间隔 < 300ms）

问题：两个 flyOut 动画同时触发，卡片叠推进逻辑冲突

解决方案：
  · 完成操作入队（FIFO），同一时刻只允许一个 flyOut 动画播放
  · 第二次点击被节流（300ms debounce），排入完成队列
  · 队列消费：前一个 flyOut animationEnd 后，自动弹出队列下一个
  · 若队列中的任务已不在主控台（被其他操作移除），跳过
```

### 18.2 等待时间精度与跨日处理

```
场景：用户设置 8h 等待，然后关闭应用过夜

问题：
  · 浏览器 tab 休眠后 setTimeout/setInterval 精度退化
  · 次日重新打开时，等待时间可能已超期数小时

解决方案：
  · 等待计算基于 waitStartedAt + waitMinutes 的绝对时间戳，不依赖 timer 累计
  · 恢复会话时：currentTime > waitEndAt → 直接标记 wait-ended
  · 环形进度条渲染：每次 CD 触发时读取 Date.now() - waitStartedAt 计算进度
  · 长时间等待（> 2h）：圆环每 30s 更新一次 dashoffset（CSS transition 30s），
    而非每秒更新，减少渲染压力
```

### 18.3 碎片阶段无日常任务且等待时间 > 2min 的处理

```
情况：碎片阶段进入条件满足，但用户未配置任何日常任务，且最近的等待 > 10min

v1.0 G-07 的降级兜底仅覆盖 ≤ 2min 场景。对于 > 2min 但无日常任务的情况：

解决方案：
  · 显示「留白期」呼吸界面（控制台整体透明度 70%，缓慢呼吸动画）
  · 中央文案："所有任务正在等待中，最近的 [任务 X] 将在 Xmin 后恢复"
  · 底部显示「新建临时任务」按钮，允许用户临时创建碎片任务填充
  · 如果用户不操作，界面保持静谧不干扰
```

### 18.4 算法推荐 3s 自动晋升的可取消性强化

```
v1.0 §7.5 定义了"无操作 3s → 最高分推荐任务自动晋升"

风险：
  · 用户可能正在阅读任务详情，3s 不操作不代表同意
  · 自动晋升破坏心流

修正：
  · 3s 进度条出现后，用户可按 Escape 或点击进度条取消
  · 取消后 30s 内不再自动推荐（grace period）
  · 用户可在设置中关闭自动晋升（默认开启）
  · 移动端：3s 进度条更醒目（底部全宽条），触碰任意区域取消
```

### 18.5 挂起任务下沉后虚拟队列位置冲突

```
场景：主控台有 3 张可见卡片 (A/B/C)，A 挂起下沉

问题：A 应该"静悄悄出现在卡片叠最后方"，但如果虚拟队列中已有 D/E

解决方案：
  · A 挂起后插入位置 = 可见叠最后一张的后面，而非虚拟队列的最后
  · 即 A 变为第 3 层可见位（25% 透明度），而非进入虚拟队列
  · 若可见叠已有 3 张（满员），A 进入虚拟队列的第一位（队头），
    状态从 focusing → suspend-waiting
  · StatusHUD 中 A 始终可见，不受可见叠上限影响
```

### 18.6 C 位卡片详情编辑状态下的挂起/完成保护

```
场景：用户正在 C 位卡片的详情文本框中打字

问题：
  · 此时另一个任务的等待到期触发自动推荐晋升，C 位可能被替换
  · 用户打字中误触 Alt+滚轮触发认知负荷切换

解决方案：
  · C 位卡片处于 input/textarea 焦点状态时，标记为 editLock = true
  · editLock = true 时：
    - 等待到期仅在 StatusHUD 提示（呼吸光晕），不自动推荐切换
    - Alt+滚轮事件不触发认知负荷切换（仅在卡片非焦点区域触发）
    - 完成按钮需二次确认（防误触：短暂出现"确认完成？"）
  · 用户离开输入框（blur）后 editLock 自动解除
```

### 18.7 跨项目任务的状态一致性

```
场景：任务 A 来自项目 X，停泊坞中标记为"专注中"；此时用户切到项目 X 的文本栏

问题：文本栏中任务 A 的状态指示与停泊坞不一致（文本栏显示"未完成"，停泊坞显示"专注中"）

解决方案：
  · 在文本栏和流程图栏中，已入坞的任务块增加视觉标记："🚢 已入坞"
  · 专注中的任务额外显示脉冲边框（复用 StatusHUD 的 indigo 脉冲色）
  · DockEngineService 维护 dockedTaskIds: Set<string> 信号，
    文本栏/流程图栏组件 computed 读取此集合决定是否显示入坞标记
  · 数据层无需同步额外字段，视觉标记纯粹在前端本地计算
```

### 18.8 日常任务槽的时区与跨日判定

```
日常任务的"每天最多显示 N 次"依赖"今天"的定义

问题：用户跨时区旅行，或凌晨 00:00~06:00 连续使用，completedDate 归属哪天？

解决方案：
  · "今天"使用用户本地时间的日期（new Date().toLocaleDateString('sv')）
  · 应用启动/恢复时检查 todayDate 是否变化，变化则重置 todayCompletedCount
  · 不依赖服务端时区，纯粹本地判断
  · 设置页提供"日常任务重置时间"选项（默认 00:00，可设为 05:00 夜猫模式）
```

### 18.9 控制台任务总数超出上限的兜底

```
v1.0 定义：雷达区(8) + 外围区(10) + 主控台(~4) = 最多 ~22 个槽位

问题：用户从多个项目持续拖入任务，可能超 22 个

解决方案：
  · 软上限 15 个：达到时显示 Toast "建议控制在 15 个以内以保持专注"
  · 硬上限 30 个：拒绝新增（除非移除现有任务），避免 UI 崩溃
  · 溢出任务（暂定 22 个以上）不在三区显示，进入"溢出队列"
  · 溢出队列在停泊坞面板底部展示为折叠列表
```

### 18.10 算法推荐与用户手动选择的冲突

```
场景：算法高亮推荐任务 B，但用户点选了任务 D

问题：B 的高亮脉冲动画仍在播放，视觉混乱

解决方案：
  · 用户点选任何非推荐任务后，立即取消所有推荐高亮（0ms，无渐消）
  · 3s 自动晋升进度条同步消失
  · 被选中的 D 直接执行磁力吸附晋升动画
  · 算法在下一次触发时重新计算（不缓存上次推荐结果）
```

### 18.11 就地创建任务的 ID 与原始 Task 关系

```
v1.0 §4.2.5 定义了就地创建规则，但未明确创建的"任务"是 FocusTaskSlot 还是真正的 Task

解决方案：
  · 就地创建 = 同时创建一个真正的 Task（通过 TaskOperationAdapterService.addTask）
  · FocusTaskSlot.taskId 指向该新 Task 的 ID
  · sourceProjectId = 当前聚焦项目 ID（若有）或 null
  · 专注会话结束时，sourceProjectId = null 的任务触发归档提示（见 §4.2.5）
  · 好处：保证所有任务都有 Task 实体，离线同步链路统一
```

### 18.12 专注模式与 Gate（大门）的交互

```
场景：用户启动应用 → Gate 展示遗留任务 → 是否允许同时显示停泊坞？

解决方案：
  · Gate 阶段：停泊坞入口（半圆触发胶囊）可见，但仅显示计数，点击无反应
  · Gate 通过（全部裁决完毕）后：停泊坞恢复完整交互
  · Gate 中被裁决为"已完成"的任务若在停泊坞中 → 自动标记完成并弹出
  · Gate 中被裁决为"已读"（但未完成）的任务若在停泊坞中 → 保持原样
```

### 18.13 多浏览器 Tab 同时打开的停泊坞一致性

```
场景：用户在两个 Tab 中打开同一应用

问题：两个 Tab 各自维护 FocusSessionState，状态分裂

解决方案（MVP）：
  · 通过 BroadcastChannel API 同步关键操作
  · 第一个 Tab 为 "leader"（通过 localStorage lock 判定）
  · 非 leader Tab 的停泊坞处于只读模式，显示 "在另一标签页中操作"
  · 用户可点击"接管控制"按钮，切换 leader 角色
```

### 18.14 动画期间的性能降级策略

```
当设备帧率连续 3 帧低于 30fps 时：

降级措施（递进）：
  Level 1 (45fps↓): 关闭组合选择区域漂浮动画，卡片静止
  Level 2 (30fps↓): 关闭 StatusHUD 脉冲/呼吸动画
  Level 3 (20fps↓): 所有动画降级为 CSS transition 0ms（瞬间切换）
  Level 4 (极端):   组合选择区域/备选区域折叠为列表模式

检测方式：
  · requestAnimationFrame 计算帧间隔
  · 滑动窗口 5 帧平均，避免单帧毛刺触发降级
  · 降级后 10s 无卡顿 → 自动恢复上一级别
```

### 18.15 停泊坞收起/展开与专注模式的状态矩阵

```
      | 停泊坞收起    | 停泊坞展开
------+---------------+-----------------
非专注 | 半圆入口可见   | 底部面板显示完整卡片列表
      | 可拖入任务     | 可拖入/新建/设主任务
------+---------------+-----------------
专注中 | 半圆入口可见   | 面板半透明(35%)悬浮恢复
      | 拖拽悬停展开   | 悬停 → 90% 不透明
      | 三区控制台可见  | 三区控制台 + 面板同时可见
      | StatusHUD 可见 | StatusHUD 可见

关键规则：
  · 专注模式进入/退出不影响停泊坞的展开/收起状态
  · 切换项目不影响上述任何状态
  · 应用关闭 → 持久化两个独立布尔值：isFocusMode, isDockExpanded
```

### 18.16 认知负荷 null（未设置）的算法处理

```
v1.0 的 computeInterludeScore() 中，cognitiveLoad === 'low' 得 20 分

问题：未设置负荷（null）的任务得 0 分，永远不会被推荐

修正：
  · null 负荷视为"中性"，得 10 分（介于高低之间）
  · UI 上 null 显示灰色侧边条（已在 §3.2.1 定义）
  · 算法注释：null 不等于高负荷，仅表示用户未评估
```

### 18.17 完成按钮的多端一致性（v3.0 修订：撤回 1s→3s）

```
桌面端：
  · C 位卡片底部「✓ 完成任务」按钮
  · 鼠标向上拖拽卡片 > 80px → 触发完成（类似划出手势）

移动端：
  · 上划 C 位卡片 > 120px → 触发完成（v1.0 §12.2 已定义）
  · 底部操作栏保留完成按钮（上划不便时可点击）

共同规则：
  · v3.0 修订：完成触发后 3s 内可撤回（底部 Toast "任务已完成，撤回？" + 按钮）
  · 撤回操作：恢复到 C 位，状态回退到 focusing
  · 超过 3s：同步队列已推送，不再允许撤回
  · 撤回逻辑在自动推进场景下保持一致（B 完成后 A 自动推进也有 3s 撤回窗口）
```

### 18.18 专注模式退出时的数据归档（v3.0 修订：跨会话持久 + 用户确认）

```
v3.0 核心修订：专注会话跨会话持久保留，用户主动确认才结束。

退出专注模式时的确认流程：
  用户点击「关闭专注模式」按钮（灯泡 SVG）：
  → 弹出确认对话框："是否结束本次专注？"
     [结束专注并保留停泊坞] → 停泊坞保留，会话状态清除
     [结束专注并清空停泊坞] → 弹出二次确认，确认后清空
     [取消] → 回到专注模式

  跨会话持久化（v3.0 新增）：
  · hasFirstBatchSelected 应跨会话持久化（与 v2.1 的"每次重置"逻辑冲突→以 v3.0 为准）
  · 应用关闭/崩溃重启 → 自动恢复上次专注会话状态
  · 用户下次打开应用 → 提示"恢复上次专注会话？"
  · 只有用户主动选择"结束专注"才会清除 session

原归档流程保持：
  1. 所有 status = 'focusing' 的任务 → 保存当前进度，保留在停泊坞
  2. 所有 status = 'suspend-waiting' 且等待尚未结束 → 保留计时器
  3. 所有 status = 'pending' → 保留在停泊坞
  4. 所有 status = 'completed' → 已经从停泊坞移除

  用户选择：
  · "退出专注，保留停泊坞" → 停泊坞回到普通模式（非专注），任务保留
  · "退出专注，清空停泊坞" → 弹出确认对话框，确认后清空
  · "退出专注，但保持虚化" → 不存在此选项（退出 = 关闭虚化 + 关闭三区）

  默认行为（点击 Escape 或"退出专注"按钮）：退出专注，保留停泊坞
```

### 19.5 透明度层级精化

v1.0/1.1 描述的"差值 20%"规则在 v2.0 中落地为精确数值：

```
C 位          1.00  ← 100% 不透明，绝对视觉焦点
第 2 层       0.42  ← 差值 0.58（故意大，拉开与 C 位的距离感）
第 3 层       0.24  ← 差值 0.18（近黑暗，背景感强）
组合选择区域内层    0.78  ← 与第 3 层差值 0.54（跨越区域，有意留白）
组合选择区域外层    0.66
备选区域        0.44  ← 与组合选择区域差值最大 0.22（体现备选的疏离感）
遮罩          0.40  ← 专注覆盖层，不参与层级
```

> 核心洞察：主控台内部的透明度「断崖式」（0.42/0.24）制造戏剧性层次感；主控台到外区的透明度「渐进式」（0.78→0.66→0.44）制造空间深度感。两种策略共存，而非统一线性递减。
